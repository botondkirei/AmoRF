User Guide

Built-in and User Defined Non-idealities

NonIdealityClass. NonIdealityClass provides a template for the non-idealities of the toolbox. To find out more about the build-in non-idealities inspect Non-ideality Classes.

User Defined Non-idealities. A user defined non-ideality class should be derived from NonIdealityClass. To keep the new class compatible with exiting functionalities, the header of process member should be the same (the input of the method is the signal that is going to be affected by the nonideality, the output is the affected signal). An example is provided in the documentation of NonIdealityClass.

Modeling an RF component

ComponentClass in the toolbox is considered to be aggregate class of non-idealities. The best practice is to instantialize objects of non-idealities and include them to a RF block model. An example is provided in ComponentClass.

Modeling an RF receiver

The amoRF toolbox provides functions for the user to "design" it's own receiver through cascading RF components. See the modeling or a Direct Conversion Receiver (DCR) in ReceiverClass.

ReceiverClass arrives with functions to compute the equivalent noise figure (or the receiver temperature) and the overall receiver nonlinearity.
